1.1 OOP Principles
▼
Abstraction
Hiding implementation details and showing only the necessary interface to the user. Achieved via abstract classes and interfaces. Abstract class can have state and partial implementation; interface (Java 8+) can have default/static methods but no state (except constants).
Encapsulation
Bundling data and methods that operate on it together, restricting direct access via access modifiers (private, protected, public, package-private). Getters/setters provide controlled access. Immutability (final fields, no setters) is the strongest form of encapsulation.
Inheritance vs Composition
"Favour composition over inheritance." Inheritance creates tight coupling — a change in parent breaks children. Composition (HAS-A) is more flexible than Inheritance (IS-A). Java only supports single inheritance of classes but multiple interface implementation. Diamond problem in multiple interface default methods resolved by explicitly overriding.
Polymorphism
Compile-time polymorphism = method overloading (same name, different params, resolved at compile time). Runtime polymorphism = method overriding (resolved at runtime via dynamic dispatch). Covariant return type: overriding method can return a subtype of the parent's return type.
SOLID Principles
SRP: one class, one reason to change. OCP: open for extension, closed for modification — use Strategy/Factory. LSP: subclass must be substitutable for parent — Square extending Rectangle violates this. ISP: don't force clients to implement unused methods — split fat interfaces. DIP: depend on abstractions not concretions — Spring DI enforces this.
Abstract class vs Interface
Abstract class: state, constructors, partial impl. Interface: contract only (Java 8+ allows default methods). Use abstract class for shared behaviour; interface for capability contracts.
Access Modifiers
private (class only) → package-private (default, same package) → protected (package + subclasses) → public (everywhere). Know the exact scope of each.
final keyword
final class = cannot extend. final method = cannot override. final variable = cannot reassign (reference; for objects the object itself is still mutable). final + static = constant.
static keyword
Belongs to class not instance. Static methods cannot access instance members. Static inner class vs inner class. Static initializer blocks.

1.2 Collections Framework Internals
▼
HashMap Internals (Java 8+)
Array of buckets (Node[]). Key's hashCode() is computed, then index = hash & (capacity-1). Collision handling: linked list. When bucket size ≥ 8 AND total capacity ≥ 64 → converts to red-black tree (O(log n) lookup). Default load factor = 0.75 → resize at 75% capacity (doubles size, rehashes all entries). Always override both equals() AND hashCode() together.
ConcurrentHashMap (Java 8)
No segment locking (Java 7 had this). Java 8 uses CAS operations + synchronized only on individual bucket heads. Reads are never locked. Writes lock only the affected bucket. computeIfAbsent(), merge() are atomic. Does NOT allow null keys or null values (unlike HashMap).
LinkedHashMap
Extends HashMap but maintains a doubly-linked list through entries. Two modes: insertion-order (default) and access-order (new LinkedHashMap(16, 0.75f, true)). Access-order makes it perfect for LRU cache — override removeEldestEntry().
TreeMap
Red-black tree. Keys sorted by natural order or custom Comparator. NavigableMap operations: floorKey(), ceilingKey(), headMap(), tailMap(). O(log n) for get/put. Not thread-safe.
ArrayList vs LinkedList
ArrayList: dynamic array, O(1) random access, O(n) insert/delete in middle, excellent cache locality. LinkedList: doubly-linked, O(1) insert/delete at known position, O(n) random access, poor cache locality (nodes scattered in memory). In practice ArrayList almost always wins. LinkedList only wins for constant add/remove at head/tail.
Fail-fast vs Fail-safe Iterators
Fail-fast: ArrayList, HashMap — throws ConcurrentModificationException if collection modified during iteration (checks modCount). Fail-safe: CopyOnWriteArrayList, ConcurrentHashMap — iterates over a snapshot or handles concurrent modification gracefully. No CME but may not see latest data.
PriorityQueue
Min-heap by default. O(log n) insert/remove, O(1) peek. Custom order via Comparator. Used in Dijkstra, top-K problems. Does NOT maintain sorted order on iteration.
ArrayDeque
Resizable array-based deque. Faster than Stack (no synchronization) and LinkedList for most stack/queue use cases. Use as stack (push/pop) or queue (offer/poll).
HashSet / TreeSet / LinkedHashSet
HashSet = HashMap with dummy values. TreeSet = TreeMap with dummy values (sorted). LinkedHashSet = LinkedHashMap with dummy values (insertion order). All backed by their Map counterparts.
Collections utility class
sort(), binarySearch(), reverse(), shuffle(), unmodifiableList(), synchronizedList(). Know what synchronizedList does NOT protect against (compound operations like check-then-act).

1.3 Java 8–17 Features
▼
Streams API
Lazy evaluation — intermediate operations (filter, map, flatMap, sorted, distinct, limit, peek) are NOT executed until a terminal operation (collect, forEach, count, reduce, findFirst) is called. Streams are consumed once. flatMap flattens a Stream<Stream<T>> into Stream<T>. Collectors: groupingBy, partitioningBy, toMap, joining, counting.
Optional
Container for a value that may or may not be present. Avoids NullPointerException. orElse(x) — x is ALWAYS evaluated even if value present (eager). orElseGet(() -> x) — x only evaluated if empty (lazy). This is a classic interview trap. Anti-patterns: Optional as method param, Optional as field, Optional.get() without check.
Lambda & Functional Interfaces
Functional interface = exactly one abstract method (can have default/static). Built-in: Function<T,R> (T→R), BiFunction<T,U,R>, Predicate<T> (T→boolean), Supplier<T> (→T), Consumer<T> (T→void), UnaryOperator<T>, BinaryOperator<T>. Method references: ClassName::method, instance::method, ClassName::new.
Records (Java 16)
Immutable data carriers. record Point(int x, int y){} auto-generates: constructor, getters (x(), y()), equals(), hashCode(), toString(). Cannot extend other classes (implicitly extends Record). Can implement interfaces. Compact constructor for validation. Ideal for DTOs.
Sealed Classes (Java 17)
Restrict which classes can extend/implement a class/interface. sealed interface Shape permits Circle, Rectangle, Triangle. Each permitted class must be final, sealed, or non-sealed. Enables exhaustive pattern matching in switch. Useful for modelling closed domain types (like payment states).
Pattern Matching & Switch Expressions (Java 14–17)
Pattern matching for instanceof: if (obj instanceof String s) { s.length(); } — no separate cast needed. Switch expressions (Java 14): return a value, use arrow syntax, no fall-through. Switch pattern matching (Java 17 preview): match by type in switch.
var (Java 10)
Local variable type inference. Only for local variables. The type IS inferred at compile time — not dynamic like JavaScript. Hurts readability in complex generic types if overused.
Text Blocks (Java 15)
Multi-line strings with """ delimiter. Automatic indentation stripping. Useful for SQL queries, JSON, HTML in code. No need to escape quotes.
Parallel Streams
Uses ForkJoinPool.commonPool(). Risks: not always faster (overhead), shared mutable state = race conditions, order not guaranteed, DB connections in thread pool context. Use only for CPU-bound operations on large data sets.
Stream.collect() deep dive
Collectors.groupingBy(key) → Map<K, List<V>>. groupingBy(key, counting()) → Map<K,Long>. groupingBy(key, summingInt(fn)) → Map<K,Integer>. toMap(key, value, mergeFunction) for handling duplicate keys.

1.4 JVM Internals & Garbage Collection
▼
JVM Memory Areas
Heap: Young Gen (Eden + S0 + S1) + Old Gen. Objects start in Eden, minor GC promotes survivors to S0/S1, after threshold → Old Gen. Stack: per-thread, holds frames (local vars, operand stack, method reference). Metaspace: class metadata (replaced PermGen in Java 8), grows dynamically, can cause OOM. Code Cache: JIT-compiled native code. Native Memory: NIO buffers, JNI.
Garbage Collectors
Serial GC: single thread, stop-the-world, for small apps. Parallel GC: multiple threads, still STW, good throughput. G1 GC (default Java 9+): divides heap into equal regions, concurrent marking, predictable pause goals, handles large heaps well. ZGC (Java 15+): sub-millisecond pauses, concurrent, scalable to TB heaps. Shenandoah: similar to ZGC. GC roots: stack references, static fields, JNI references.
Java Memory Model (JMM)
Defines how threads interact through memory. Each thread has its own working memory (CPU cache). Happens-before guarantee: if A happens-before B, A's effects are visible to B. synchronized establishes happens-before on monitor. volatile write happens-before volatile read of same variable. Without happens-before, the JVM/CPU can reorder operations and cache values — leading to visibility bugs.
Memory Leaks in Java
Java has GC but leaks still happen when objects are reachable but logically unused. Common causes: static collections holding references, unremoved event listeners/callbacks, unclosed resources (Connection, Stream), ThreadLocal not cleaned in thread pool, intern() abuse, inner class holding reference to outer class, caches without eviction.
String Pool & Interning
String literals are automatically interned into the String Pool (in Metaspace since Java 8). "hello" == "hello" is true (same pool object). new String("hello") == "hello" is false (new heap object). s.intern() moves string to pool or returns existing. Overusing intern() on large unique strings bloats Metaspace.
Class Loading
Bootstrap (JDK classes) → Extension → Application classloader. Delegation model: always ask parent first. Dynamic class loading: Class.forName(). Custom classloaders for plugins, hot-reload. ClassNotFoundException vs NoClassDefFoundError.
JIT Compilation
JVM interprets bytecode first, then JIT-compiles hot methods to native code. C1 (client) = fast compile, less optimised. C2 (server) = slow compile, heavily optimised. Tiered compilation uses both.
WeakReference / SoftReference
WeakReference: GC'd in next collection (use for WeakHashMap caches). SoftReference: GC'd only when memory is low (use for image caches). PhantomReference: post-GC cleanup. All weaker than strong references.
OutOfMemoryError types
Java heap space (heap full), GC overhead limit exceeded (GC running >98% time), Metaspace (class metadata), unable to create native thread (OS limit), Direct buffer memory (NIO).

1.5 Exception Handling
▼
Exception Hierarchy
Throwable → Error (JVM errors, don't catch: OutOfMemoryError, StackOverflowError) | Exception → RuntimeException (unchecked) | CheckedException (checked). Checked exceptions must be declared or caught. Unchecked (RuntimeException) don't need to be declared.
Checked vs Unchecked
Checked: forces callers to handle (IOException, SQLException). Overused checked exceptions pollute APIs. Spring uses unchecked exceptions exclusively for this reason. Unchecked: RuntimeException and subclasses. Best practice: use unchecked for programming errors; checked only when caller can reasonably recover.
try-with-resources
Resources implementing AutoCloseable are closed automatically in reverse order. If both try block and close() throw, the close() exception is suppressed (accessible via getSuppressed()). If try block and finally both throw, the finally exception SWALLOWS the try exception — classic bug.
Exception chaining
Wrapping a lower-level exception in a higher-level one: throw new ServiceException("Payment failed", e). Preserves the original cause in the stack trace. Never log and rethrow — log OR rethrow, not both (causes duplicate logs).
Custom Exceptions
Extend RuntimeException for unchecked. Include error code field. Define serialVersionUID. Add constructors that accept message + cause. Name clearly: PaymentFailedException, UserNotFoundException.
Multi-catch (Java 7)
catch (IOException | SQLException e) — catches multiple types. The caught exception is effectively final — can't reassign. Can't catch exceptions in same hierarchy this way.

1.6 Generics & Type System
▼
Generics & Type Erasure
Generics exist at compile time only — erased to raw types at runtime (type erasure). List<String> and List<Integer> are both List at runtime. Cannot do new T(), new T[], or instanceof T at runtime. Wildcards: ? extends T (upper-bounded, read-only), ? super T (lower-bounded, write-only). PECS: Producer Extends, Consumer Super.
Comparable vs Comparator
Comparable: natural ordering, class implements it, compareTo(). Comparator: external ordering, passed to sort/TreeSet/PriorityQueue. Java 8: Comparator.comparing(Person::getName).thenComparing(Person::getAge). Collections.sort() uses Comparable; explicit sort uses Comparator.

2.1 Thread Fundamentals
▼
Thread Lifecycle States
NEW (created, not started) → RUNNABLE (running or ready to run) → BLOCKED (waiting for monitor lock) → WAITING (indefinitely waiting: Object.wait(), Thread.join(), LockSupport.park()) → TIMED_WAITING (waiting with timeout: sleep(), wait(ms), join(ms)) → TERMINATED (finished). BLOCKED vs WAITING: BLOCKED = waiting for lock; WAITING = voluntarily giving up CPU.
Creating Threads
4 ways: (1) extend Thread — bad, couples task to execution mechanism. (2) implement Runnable — better, no return value. (3) implement Callable — returns Future, can throw checked exceptions. (4) ExecutorService.submit() — best practice, decouples task from thread management. Never create raw threads in production code — always use ExecutorService.
Daemon Threads
JVM exits when all non-daemon threads finish, even if daemon threads are still running. GC thread, JIT compiler thread are daemon. Set with thread.setDaemon(true) BEFORE start(). Use for background tasks that should not prevent JVM shutdown (logging, monitoring). Don't use for tasks that must complete (DB writes).
Thread Interruption
Cooperative mechanism. thread.interrupt() sets the interrupt flag. The thread must CHECK this flag (isInterrupted()) or be in a blocking operation that throws InterruptedException. When InterruptedException is caught, the flag is cleared — you must re-interrupt: Thread.currentThread().interrupt(). Never swallow InterruptedException silently.

2.2 Synchronisation & Locks
▼
synchronized keyword
Every object has an intrinsic monitor lock. synchronized(obj) acquires obj's lock. synchronized method acquires this's lock (instance method) or ClassName.class's lock (static method). Reentrant: same thread can acquire same lock multiple times. Establishes happens-before: unlock happens-before subsequent lock of same monitor.
volatile keyword
Guarantees VISIBILITY only — reads go to main memory, writes flush to main memory. Does NOT guarantee atomicity. volatile int count; count++; is NOT thread-safe (count++ is read-modify-write = 3 operations). Use volatile for: flags (boolean stop = false), double-checked locking (the field MUST be volatile), publishing safely constructed objects.
ReentrantLock
More flexible than synchronized. tryLock() — non-blocking attempt. lockInterruptibly() — can be interrupted while waiting. fairness — longest-waiting thread gets lock first (default false = unfair). ALWAYS unlock in finally block: try { lock.lock(); ... } finally { lock.unlock(); }. ReentrantReadWriteLock: multiple concurrent readers OR single writer.
Atomic Classes
java.util.concurrent.atomic package. AtomicInteger, AtomicLong, AtomicBoolean, AtomicReference. Use CAS (Compare-And-Swap) hardware instruction — lock-free, no blocking. compareAndSet(expected, update) — atomically sets to update only if current == expected. Handles counter, flags, reference updates without locks. Faster than synchronized for simple operations.
Deadlock
4 necessary conditions (Coffman): Mutual exclusion, Hold and wait, No preemption, Circular wait. Prevention: always acquire locks in same order; use tryLock() with timeout; avoid nested locks. Detection: thread dump shows threads in BLOCKED state waiting for each other's locks. jstack/VisualVM shows "Found one Java-level deadlock".
StampedLock (Java 8)
Better than ReadWriteLock for read-heavy workloads. Optimistic read: long stamp = lock.tryOptimisticRead() — no lock acquired. Then validate: lock.validate(stamp) — if false, upgrade to read lock. Write lock invalidates all optimistic reads. Faster when reads rarely conflict with writes.

2.3 ExecutorService & Thread Pools
▼
ThreadPoolExecutor 7 Parameters
corePoolSize: minimum threads always alive. maximumPoolSize: max threads ever. keepAliveTime: extra threads idle timeout. unit: time unit. workQueue: task queue (LinkedBlockingQueue = unbounded! SynchronousQueue = no buffer). threadFactory: name threads for debugging. rejectedExecutionHandler: AbortPolicy (throw), CallerRunsPolicy (caller executes), DiscardPolicy (silently drop), DiscardOldestPolicy. Thread creation: if < core → new thread; if ≥ core → queue; if queue full AND < max → new thread; if queue full AND = max → reject.
Executors factory methods — pitfalls

ForkJoinPool
Work-stealing: idle threads steal tasks from busy threads' deques. Used by parallel streams and CompletableFuture's default pool. ForkJoinPool.commonPool() is shared JVM-wide — using it for blocking IO is dangerous. RecursiveTask (returns result) vs RecursiveAction (no result). Best for divide-and-conquer CPU-bound work.

2.4 CompletableFuture
▼
Core Methods
supplyAsync(supplier): runs in ForkJoinPool, returns value. runAsync(runnable): no return. thenApply(fn): transform result synchronously (like map). thenCompose(fn): chain another CompletableFuture (like flatMap — use when fn returns CF). thenCombine(cf, fn): combine two independent CFs. allOf(cfs): wait for ALL. anyOf(cfs): complete when ANY completes. exceptionally(fn): recover from exception. handle(fn): runs regardless of success/failure.
thenApply vs thenCompose
thenApply: CF<T> → T → R → CF<R>. Use for synchronous transformation. thenCompose: CF<T> → T → CF<R> → CF<R>. Use when your function itself returns a CompletableFuture (avoids CF<CF<R>>). Rule: if your function returns a plain value → thenApply. If it returns a CF → thenCompose.
Exception Handling
exceptionally(ex -> fallback): only called on failure, returns a value. handle((result, ex) -> ...): always called, handles both success and failure. whenComplete((result, ex) -> ...): always called for side effects, does NOT transform result. If exception is not handled, it's wrapped in ExecutionException when you call get().

2.5 Concurrency Utilities
▼
CountDownLatch
One-time barrier. Initialised with count N. latch.await() blocks until count reaches 0. Workers call latch.countDown(). Cannot be reset. Use: wait for N services to initialise before starting; wait for N tasks to complete.
CyclicBarrier
Reusable barrier. All N parties call barrier.await() and block until all N have arrived. Optional Runnable executes when barrier trips. Resets for next cycle. Use: phased computations where all threads must complete phase 1 before any starts phase 2.
Semaphore
Controls concurrent access to a resource pool. semaphore.acquire() decrements permit count (blocks if 0). semaphore.release() increments. Use: limit concurrent connections to a resource, rate limiting, connection pooling logic.
BlockingQueue
Thread-safe producer-consumer queue. put() blocks if full. take() blocks if empty. offer(e, timeout) / poll(timeout) with timeout. ArrayBlockingQueue (bounded), LinkedBlockingQueue (optionally bounded), PriorityBlockingQueue (ordered), SynchronousQueue (zero capacity — direct handoff).
ThreadLocal
Per-thread storage. Each thread has its own independent copy. Spring's SecurityContextHolder, RequestContextHolder use ThreadLocal. ▼

3.3 @Transactional Deep Dive
▼
Propagation Types (Know all 7)
REQUIRED (default): join existing or create new. REQUIRES_NEW: always create new, suspend existing. NESTED: savepoint within existing (only works with JDBC; if inner fails, rolls back to savepoint, outer continues). SUPPORTS: join if exists, else no transaction. NOT_SUPPORTED: suspend existing, run without. MANDATORY: must have existing, throw if none. NEVER: must not have existing, throw if exists.
Isolation Levels
READ_UNCOMMITTED: can read uncommitted data (dirty read). READ_COMMITTED (PostgreSQL default): no dirty reads, allows non-repeatable reads. REPEATABLE_READ (MySQL default): no dirty/non-repeatable reads, allows phantom reads. SERIALIZABLE: full isolation, lowest concurrency. Anomalies: dirty read (read uncommitted data that gets rolled back), non-repeatable read (same row read twice gives different values), phantom read (range query returns different rows on second read).
Rollback Rules — The Hidden Trap
By default, @Transactional ONLY rolls back on RuntimeException and Error. Checked exceptions do NOT trigger rollback by default. This catches many teams off guard. Fix: @Transactional(rollbackFor = Exception.class) or @Transactional(rollbackFor = {IOException.class, MyCheckedException.class}). Also: noRollbackFor to exclude specific unchecked exceptions.
readOnly = true
Hints to the persistence provider and transaction manager. Hibernate sets FlushMode to NEVER (no dirty checking, faster). Some JDBC drivers and connection pools route to read replicas. Does NOT guarantee that no writes happen — it's a hint, not enforcement. Always use for queries.

4.2 JWT Authentication
▼
JWT Structure
3 parts: Header (algorithm, type) . Payload (claims: sub, iat, exp, custom) . Signature. All Base64URL encoded. Header + Payload are NOT encrypted (only signed) — anyone can decode. NEVER store sensitive data (password, SSN) in JWT. HS256: symmetric (shared secret, fast). RS256: asymmetric (private key signs, public key verifies — use when multiple services need to verify without sharing secret).
Access + Refresh Token Pattern
Access token: short-lived (15min–1hr), stateless JWT, sent in Authorization: Bearer header. Refresh token: long-lived (days–weeks), stored in HttpOnly cookie or DB, used to get new access token. Refresh token rotation: issue new refresh token with every use, invalidate old one (detects theft). On logout: blacklist access token (Redis) until expiry, invalidate refresh token in DB.
Token Revocation Challenge
JWT is stateless — server has no state to invalidate. Options: (1) Short expiry (accept short window of misuse). (2) Blacklist in Redis: on logout, add token JTI (JWT ID) to Redis with TTL = token expiry. On each request, check blacklist (O(1) Redis lookup). (3) Opaque tokens with introspection: token is a random string, server does DB lookup on every request (slower but fully revocable).

▼
OAuth 2.0 Flows

5.1 JPA/Hibernate Internals
▼
Entity Lifecycle States
Transient: not associated with session, not in DB. Managed/Persistent: associated with active EntityManager, changes tracked (dirty checking). Detached: was managed, EntityManager closed or explicitly detached — changes NOT tracked. Removed: scheduled for deletion. Methods: persist() (transient→managed), merge() (detached→managed, returns new managed instance), detach(), remove(), refresh().
N+1 Problem
For 1 query that returns N entities, Hibernate fires N additional queries to load lazy associations. Example: fetch 100 Orders → then 100 separate queries to fetch each Order's Customer. Fix: (1) JOIN FETCH in JPQL: SELECT o FROM Order o JOIN FETCH o.customer. (2) @EntityGraph: declarative way to eager-load specific associations. (3) Batch fetching: @BatchSize(size=20) fetches 20 at a time. (4) DTO projection with native query.
Dirty Checking & Flush
Hibernate takes a snapshot of managed entities at load time. On flush (before query, before commit, explicit EntityManager.flush()), it compares current state to snapshot and generates UPDATE SQL for changed fields. @DynamicUpdate: only updates changed columns (useful for wide tables). FlushMode: AUTO (before queries and commit), COMMIT (only before commit), MANUAL.
Optimistic vs Pessimistic Locking
Optimistic: assume no conflict, check at commit. @Version field (int or timestamp). On conflict: OptimisticLockException / StaleObjectStateException. Retry the operation. Best for low-contention scenarios (most reads, occasional writes). Pessimistic: lock at read time. LockModeType.PESSIMISTIC_WRITE (SELECT FOR UPDATE). Blocks other writers. Best for high-contention critical sections (payment processing, inventory decrement).
LazyInitializationException
Occurs when accessing a lazy collection/association AFTER the EntityManager/Session is closed. Common in Spring: entity loaded in @Transactional service, returned to controller, controller accesses lazy field — session already closed. Fixes: (1) Use DTO projection (best). (2) JOIN FETCH / @EntityGraph in query. (3) @Transactional on controller (bad — long transactions). (4) Open Session in View filter (anti-pattern — lazy queries in view layer).

6.1 Core Patterns
▼
API Gateway

Circuit Breaker (Resilience4j)
3 states: CLOSED (requests flow, failures counted) → OPEN (requests fail fast, no calls made, saves downstream) → HALF_OPEN (probe requests sent, if success → CLOSED, if fail → OPEN). Configuration: failureRateThreshold (e.g. 50%), slidingWindowSize, waitDurationInOpenState, permittedCallsInHalfOpenState. @CircuitBreaker(name="payment", fallbackMethod="paymentFallback").
Saga Pattern
Manage distributed transactions across services without 2PC. Choreography: each service publishes event, next service listens and acts. No central coordinator. Pros: loose coupling. Cons: hard to track, complex error flows. Orchestration: central saga orchestrator (Camunda, Spring State Machine) tells each service what to do. Pros: visible flow, easier error handling. Cons: central point of coupling. Compensation transactions: rollback equivalent for each step (e.g. refund if notification fails after payment).
Bulkhead Pattern
Isolate failures. Thread pool per downstream service: if ServiceB is slow, it only exhausts ServiceB's thread pool, not the shared pool. Resilience4j @Bulkhead supports thread pool isolation and semaphore isolation. Prevents one slow service from cascading failures to all other service calls.
Strangler Fig Pattern
Incrementally migrate a monolith. New features go into microservices. A routing layer (API Gateway or proxy) routes: new features → microservice, legacy features → monolith. Gradually strangle the monolith by migrating feature by feature. You used this conceptually with the Whirlpool legacy system.

7.2 Scalability Fundamentals
▼
Horizontal vs Vertical Scaling
Vertical: add more CPU/RAM to existing server. Simple but has ceiling and single point of failure. Horizontal: add more servers. Requires stateless services (no local state that can't be shared). Session stickiness problem: user's requests must hit same server if state is local. Fix: externalise session to Redis.
Load Balancing
L4 (TCP level): fast, no HTTP awareness, simple round-robin. L7 (HTTP level): routing by path/header, SSL termination, content-based routing. Algorithms: round-robin, weighted round-robin, least connections (for variable-duration requests), IP hash (sticky sessions). Health checks remove unhealthy instances from rotation.
Consistent Hashing
Hash ring with virtual nodes. Map keys to nodes. Adding/removing a node only remaps ~1/N of keys (vs naive mod-N which remaps everything). Used in: distributed caches (Redis Cluster), sharding, CDN. Virtual nodes: each physical node owns multiple positions on ring for better load distribution.
CAP Theorem
Consistency (all nodes see same data), Availability (every request gets a response), Partition Tolerance (system works despite network splits). Can only fully guarantee 2 of 3 DURING a partition. CP (choose consistency): RDBMS, ZooKeeper, HBase. AP (choose availability): Cassandra, DynamoDB, Couchbase. Kafka: AP for producers (acks=1), CP for acks=all. In practice, partition tolerance is required — so real choice is C vs A during failures.

7.3 Caching Strategy
▼
Cache Strategies
Cache-aside (most common): app checks cache first, on miss loads from DB and populates cache. App controls everything. Read-through: cache sits in front of DB, handles miss automatically. Write-through: writes go to cache AND DB synchronously (consistent but slower writes). Write-behind: writes go to cache, async DB write (fast but risk of data loss on crash). Refresh-ahead: proactively refresh before expiry.
Cache Problems
Cache penetration: query for non-existent key bypasses cache every time → DB overloaded. Fix: cache null/empty results with short TTL, or Bloom filter. Cache avalanche: many keys expire at same time → thundering herd on DB. Fix: add random jitter to TTLs. Cache stampede: popular key expires, many requests simultaneously try to rebuild it. Fix: probabilistic early expiration, distributed lock on cache rebuild, background refresh.

8.1 Architecture Internals
▼
Core Concepts
Kafka is a distributed, partitioned, replicated commit log. Topic: logical stream. Partition: ordered, immutable sequence of records. Each record has an offset. Broker: server that stores partitions. Producer: writes to topic. Consumer Group: each partition assigned to exactly one consumer in a group. More consumers than partitions → some consumers idle. More partitions than consumers → one consumer handles multiple partitions.
Replication & ISR
Each partition has a leader and N followers. Leader handles all reads/writes. Followers replicate. ISR (In-Sync Replicas): set of replicas fully caught up to leader. min.insync.replicas=2: producer requires acks from at least 2 replicas. Unclean leader election: allow out-of-sync replica to become leader (risks data loss, but improves availability). Default is false (data safety first).
Log Compaction
Instead of time/size retention, keep LATEST record for each key. Previous versions compacted away. Use for: changelog topics (latest user state), CDC (change data capture), maintaining materialised views. Regular topics are append-only and expire old records. Compacted topics are like a key-value store in Kafka.

8.2 Producers & Delivery Semantics
▼
acks Setting
acks=0: fire-and-forget, no durability guarantee. acks=1: leader acks, followers may not have it yet — can lose data if leader fails before replication. acks=all (-1): leader waits for all ISR to ack — strongest durability, higher latency. Combined with min.insync.replicas=2: at least 2 replicas must ack.
Idempotent Producer
enable.idempotence=true: each message gets a sequence number + producer ID. Kafka deduplicates retries within a partition. Forces acks=all, max.in.flight.requests.per.connection≤5, retries>0. Guarantees exactly-once within a single partition per producer session.
Exactly-Once Semantics (EOS)
Full EOS requires: idempotent producer + transactional API + consumer with isolation.level=read_committed. Producer wraps operations in transactions: beginTransaction → send messages → commitTransaction (or abortTransaction). Consumer with read_committed only sees committed messages. Complex to implement — use Kafka Streams or Spring Kafka's @Transactional support.
Partitioning Strategy
With key: hash(key) % numPartitions — same key always goes to same partition (ordering guarantee per key). Null key: sticky partitioner (fills a batch to one partition, then switches — reduces requests). Custom partitioner: implement Partitioner interface. Ordering: Kafka only guarantees order WITHIN a partition. If you need all messages for a user ordered → use userId as key.

8.3 Consumers & Error Handling
▼
Consumer Offset Management
Auto-commit (enable.auto.commit=true): commits every auto.commit.interval.ms — can duplicate on crash (processed but not committed) or lose (committed before processing). Manual commit: consumer.commitSync() (blocks, retries), consumer.commitAsync() (non-blocking, no retry). At-least-once: process then commit. At-most-once: commit then process (not recommended).
Consumer Rebalancing
Triggered when: consumer joins/leaves group, consumer crashes (session.timeout.ms exceeded), topic partition count changes. Eager rebalance: all consumers stop, all partitions reassigned. Cooperative sticky rebalance (default Kafka 3+): only moved partitions are revoked, others continue. Settings: session.timeout.ms (default 45s) — heartbeat must arrive within this. max.poll.interval.ms (default 5min) — time between polls — if exceeded, consumer considered dead.

Failed messages → retry topics (with exponential backoff) → DLT (if all retries exhausted). Spring Kafka: @RetryableTopic(attempts=3, backoff=@Backoff(delay=1000, multiplier=2)) — creates retry-topic-0, retry-topic-1000, retry-topic-2000 automatically. @DltHandler for final dead letter processing. 8.4 KSQL / ksqlDB & Avro (Your Real Experience)
▼
ksqlDB Stream vs Table
Stream: unbounded sequence of events (append-only). Each record is an independent event. Like a Kafka topic with schema. Table: materialised view of latest state per key. Compacted topic semantics. Queries on tables give latest state. Joins: stream-stream (windowed), stream-table (enrich event with current state), table-table. Persistent queries run continuously.
Avro & Schema Registry
Avro: binary serialisation format. Compact (vs JSON). Schema embedded once in registry, payload contains schema ID (4 bytes) + serialised data. Schema Registry: centralised schema storage, schema evolution management. Producer registers schema, consumer fetches by ID. Compatibility: BACKWARD (new can read old), FORWARD (old can read new), FULL (both), NONE. Default: BACKWARD.
Schema Evolution Rules
BACKWARD compatible changes: add optional field with default, remove field. BACKWARD incompatible: add required field without default (old consumer can't deserialize), change field type, rename field (use aliases). Best practice: always add defaults to new fields. Never rename or change types of existing fields in production schemas.

9.1 Creational Patterns
▼
Singleton (Thread-safe)
Several implementations: (1) Eager init: private static final Singleton INSTANCE = new Singleton(); — simple, thread-safe, created even if never used. (2) Double-checked locking: if (instance == null) { synchronized(this) { if (instance == null) { instance = new Singleton(); }}} — the field MUST be volatile (prevents instruction reordering where half-constructed object is visible). (3) Bill Pugh / Holder pattern: inner static class initialised lazily by JVM. (4) Enum: the best way — JVM guarantees single instance, serialisation-safe.
Factory Method vs Abstract Factory
Factory Method: defines interface for creating object, subclasses decide what to instantiate. Use when you have a class that can't anticipate the type it needs to create. Abstract Factory: creates FAMILIES of related objects. Use when system must be independent of how its objects are created. Example: UI toolkit creating buttons, checkboxes, menus — all consistent for a theme.
Builder Pattern
Construct complex objects step by step. Avoids telescoping constructors (many optional params). Returns the builder from each setter (fluent API). build() validates and creates the object. Lombok @Builder generates this automatically. Use for: objects with many optional fields (>3-4), immutable objects, objects that require validation before creation.
Prototype Pattern
Clone an existing object instead of creating from scratch. Java Cloneable + clone() is broken (returns Object, shallow by default, clone() is protected). Better: copy constructor, copy factory method, serialisation-based deep copy, or libraries like ModelMapper. Spring's prototype scope is named after this pattern.

10.1 REST Principles & HTTP
▼
REST Constraints
6 constraints: Statelessness (no session on server, each request self-contained), Uniform Interface (resource identification via URI, manipulation via representations), Client-Server separation, Layered System (client doesn't know about intermediaries), Cacheable responses, Code on Demand (optional — JS delivery). Statelessness is the most important for scalability.
HTTP Methods & Idempotency
Idempotent: GET, HEAD, PUT, DELETE (same result no matter how many times called). NOT idempotent: POST (creates new resource each time), PATCH (may not be idempotent depending on operation). Safe: GET, HEAD (no side effects). Idempotency matters for retries — retrying a PUT is safe; retrying a POST may create duplicates.
HTTP Status Codes (Know the nuances)
200 OK, 201 Created (POST), 204 No Content (DELETE/PUT with no body). 400 Bad Request (malformed), 401 Unauthorized (not authenticated), 403 Forbidden (authenticated but not authorized), 404 Not Found, 409 Conflict (duplicate, state conflict), 422 Unprocessable Entity (syntactically valid but semantically wrong — validation errors), 429 Too Many Requests. 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout.
API Versioning
URI versioning: /v1/users (most common, most explicit, breaks REST purity). Header versioning: Accept: application/vnd.company.v1+json (clean URLs, harder to test). Query param: /users?version=1 (easy but pollutes params). No versioning + evolve carefully (HATEOAS ideal). Trade-offs: URI is most practical; header is most RESTful.
Pagination
Offset pagination: ?page=2&size=20. Simple but unstable for changing data (inserts/deletes shift results). Cursor/keyset pagination: ?after=encodedCursor. Cursor encodes last seen ID or sort key. Stable, scalable, works with any page size. Use offset for simple admin UIs; cursor for infinite scroll / large datasets.

13.2 Distributed Lock with Redis
▼
Simple Distributed Lock
SET lockKey uniqueValue NX PX 30000: SET if Not eXists, with 30-second expiry (PX = milliseconds). uniqueValue = UUID (so only the holder can release). Release: Lua script (atomic check-and-delete): if redis.call('get',KEYS[1]) == ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end. Why Lua? GET then DEL is NOT atomic — another process could grab lock between the two commands.
Redlock Algorithm
For multi-node Redis where single-node lock is insufficient. Acquire lock on N/2+1 independent Redis instances. Lock is valid if acquired on majority within valid time. On success, effective TTL = min TTL minus elapsed time. Complex to implement correctly. Controversial — Martin Kleppmann argued it's not safe. For most applications, single Redis node lock is sufficient. Use Redlock only for true multi-DC scenarios.

14.1 Core Concepts
▼
Index, Document, Mapping
Index = database. Document = row (JSON). Mapping = schema (field types). Dynamic mapping: ES infers types automatically (convenient but can go wrong — numbers inferred as long, dates as text). Explicit mapping: define all field types upfront. text field: analysed, full-text searchable, not sortable. keyword field: not analysed, exact match, sortable, aggregatable. Shards & Replicas
Primary shard count: FIXED at index creation (cannot change later without reindex). This is critical — under-shard and you can't scale; over-shard and you waste memory. Each shard is a Lucene index. Replicas: copies of primary shards, on different nodes. Read requests can go to any replica. Replicas increase read throughput AND provide HA. Rule of thumb: aim for shard size 10–50GB.
Query DSL
Two contexts: query context (relevance scoring) vs filter context (yes/no, cached, faster). Bool query: must (AND, contributes to score), filter (AND, no score — use for date ranges, terms), should (OR, boosts score), must_not (NOT). Always put non-scoring conditions in filter context for performance. match vs term: match analyses input; term does exact match on keyword field.
Aggregations
Bucket aggregations: group documents (terms, date_histogram, range, filters). Metric aggregations: compute values (avg, sum, min, max, cardinality, percentiles). Pipeline aggregations: operate on other aggregation outputs. Nested: bucket inside metric inside bucket. Used heavily in your Citi reports work — this is exactly what Elasticsearch aggregations are for.

18.1 Window Functions

Window Function Syntax
function() OVER (PARTITION BY col ORDER BY col ROWS/RANGE BETWEEN ... AND ...). PARTITION BY: group rows (like GROUP BY but doesn't collapse). ORDER BY: order within partition. Frame: ROWS (physical rows), RANGE (logical range based on value). Without frame: default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.
Ranking Functions
ROW_NUMBER(): unique sequential number per row, no ties. RANK(): ties get same rank, next rank skips (1,1,3). DENSE_RANK(): ties get same rank, next rank does NOT skip (1,1,2). NTILE(n): divide into n buckets. Use ROW_NUMBER for "top N per group", RANK/DENSE_RANK for "rank within group."
Analytic Functions
LAG(col, n, default): value from n rows BEFORE current row. LEAD(col, n, default): value from n rows AFTER. FIRST_VALUE(col): first value in frame. LAST_VALUE(col): last value in frame (be careful with default frame). SUM/AVG/COUNT/MIN/MAX OVER: running total, moving average.
